## PART 1 — Must fix before you take real money

### 1.1 Apply the security migration to the live database — CRITICAL
This is the big one. The file
`supabase/migrations/20260613000000_security_phase1.sql` fixes six separate
problems, and it has to be **run in the Supabase SQL editor** against your live
project. It closes:

- **`profiles` has row‑level security switched OFF.** In `schema.sql` this line
  is live: `alter table public.profiles disable row level security;` (with a
  `-- TEMP: Disabled RLS for testing` comment). While it's off, **anyone with
  your public key — which ships in every browser — can read or edit or delete
  every user's profile**, including names and phone numbers. This is a UK
  GDPR‑reportable breach the moment you have real users. The migration turns
  RLS back on.
- **`bookings` was directly writable by the browser.** A carrier could PATCH
  their own booking to `completed` and trick the payout job into paying them
  with no real charge. The migration revokes client writes; all legitimate
  writes already go through trusted server functions.
- **Phone numbers leaked** to any logged‑in user. Migration hides them.
- **Carrier Stripe account IDs and earnings leaked** to any user. Migration
  hides them.
- **Handover photos were world‑readable** — any logged‑in user could view any
  booking's pickup/delivery photos. Migration scopes reads to that booking's
  two parties.
- **Message‑sender spoofing** — a bug in the message policy let a participant
  post a message *as the other person*. Migration fixes the check.

**Action:** run that migration file, on a Supabase branch first if you can, then
production. The file has step‑by‑step warnings and recovery notes at the bottom.
It must ship together with the `confirm-delivery.ts` change (already in the
repo) or confirming a delivery will break.

> ⚠️ **Two "fixes" from the pasted checklist you gave me were WRONG — do not use
> them literally:**
> - *"Add check `auth.uid() = auth.uid()`"* — that is always true and fixes
>   nothing. The migration uses the correct check (`sender_id = auth.uid()`).
> - *"Change to `auth.users.select('email')`"* — a browser cannot query
>   `auth.users` at all. I fixed this properly instead (see 1.3).

### 1.2 Confirm your LIVE Stripe keys aren't already in production — CRITICAL to verify
Your local `.env.local` uses **test** keys (`sk_test` / `pk_test`), which is
good. But the live site runs on Netlify/Vercel, which have their **own**
environment variables that override the local file. I could not see those from
here.

**Action (needs you logged in):** in Netlify → Site configuration →
Environment variables (and Vercel → Settings → Environment variables if you use
it), check `STRIPE_SECRET_KEY`. If it starts with `sk_live`, the app is already
able to move real money — so item 1.1 becomes an emergency. If it starts with
`sk_test`, you have breathing room, but still fix 1.1 before switching to live.

### 1.3 Broken email lookup in "Accept job" — FIXED in this session
`AcceptJobButton` queried `profiles.select('...email')`, but `profiles` has no
`email` column, so the sender never got the "your job was accepted" email. I
changed it to pass the sender's user ID and resolve the email **server‑side**
(emails live in `auth.users`, which the browser must never read). TypeScript
check passes. This is committed to your working files but still needs to be
pushed/deployed (Part 4).

### 1.4 Dependency vulnerabilities in Next.js — HIGH
`npm audit` reports Next.js 14.2.18 has 9 published advisories (one rated
**critical**), covering denial‑of‑service, cache‑poisoning, and cross‑site
scripting issues. These are framework‑level, not your code.

**Action:** update to the latest patch in the 14.2.x line and re‑test, e.g.
`npm install next@14.2 && npm run build && npm run typecheck`. Fully clearing
every advisory may eventually require moving to Next 15/16 (a bigger job) — but
the patch update closes the worst of it with minimal risk. Do this on a branch
and confirm the site still builds before deploying.

## PART 2 — What's already good (so you know where NOT to spend effort)

- **Payment amounts are computed server‑side** from the agreed price; the
  browser never tells the server how much to charge.
- **Stripe webhooks verify signatures** against your signing secret(s) and are
  **idempotent** (a `stripe_events` table stops the same event being processed
  twice). Payout transfers have a double‑pay guard via `transfer_group` lookup
  plus a `funds_released_at` DB check.
- **The escrow window is enforced.** A previous migration removed a trigger that
  used to skip the 24‑hour dispute window — good catch, already fixed.
- **PINs are stored as SHA‑256 hashes**, not plaintext. A carrier reading the
  booking row over the API sees only a hash and cannot self‑confirm a delivery.
  Handoff runs through `SECURITY DEFINER` database functions that re‑check who
  the caller is.
- **Secrets are handled correctly.** No real API keys in git history,
  `.env.local` is gitignored, and the powerful service‑role key is never
  imported into any browser (`'use client'`) file.
- **Admin area is properly gated.** Every `/admin` page, the admin API route,
  and the resolve‑dispute action check an email allowlist server‑side.
- **The cron endpoints require a secret** (`CRON_SECRET` bearer token) before
  they'll cancel jobs or release payouts.

## PART 3 — Should fix soon (not blocking, but real)

### 3.1 Dead "DO NOT USE" code still wired in — MEDIUM (foot‑gun)
Two files are marked `⚠️ DEPRECATED — DO NOT USE` but are still imported by live
components:

- `src/lib/actions/verify-pin.ts` (old plaintext‑PIN flow) ← imported by
  `components/PinVerify.tsx`
- `src/lib/actions/raise-dispute.ts` (old flow) ← imported by
  `components/DisputeForm.tsx`

The good news: the *pages* users actually reach use the safe v3 paths
(`BookingPins` → the hashed‑PIN RPCs; `DisputeButton` →
`raise-dispute-with-email`). The deprecated components appear to be unrendered.
But leaving loaded guns on the table is how someone re‑introduces a hole later.
**Action:** delete `verify-pin.ts`, `raise-dispute.ts`, `PinVerify.tsx`, and
`DisputeForm.tsx` once you've confirmed nothing renders them.

### 3.2 Security headers — LOW (already configured, just verify)
Both `netlify.toml` and `vercel.json` now set sensible headers (HSTS,
X‑Frame‑Options, X‑Content‑Type‑Options, Referrer‑Policy, Permissions‑Policy).
**Action:** after deploy, open the live site in Chrome → DevTools → Network →
click the document request → confirm the response headers are actually present
(platform config sometimes silently doesn't apply).

### 3.3 No rate limiting — LOW/MEDIUM
The waitlist and payment‑intent endpoints have no rate limiting, so they can be
hammered (spam signups, or probing). Not urgent at your volume, but worth adding
(e.g. Upstash rate limit, or Vercel/Netlify edge limits) before any launch push.

### 3.4 Marketing copy over‑promises vs. reality — MEDIUM (legal/trust)
The live homepage states as fact: "Every carrier verifies government ID before
they can accept jobs" and payments are "held securely by Stripe." Per your own
README, ID verification is currently a **one‑click demo stub** and live payouts
aren't switched on. Shipping claims you can't yet back is a consumer‑protection
and trust risk. **Action:** soften the copy to match what's actually live, or
finish the real ID verification before you drive real signups.

### 3.5 Legal pages are placeholders — MEDIUM (unchanged)
`privacy/` and `terms/` are placeholder pages. For a peer‑to‑peer marketplace
handling real parcels and real money, get a UK lawyer to review the T&Cs and
privacy policy before taking real users. (The company is registered — no.
17274629 is shown in the footer — so real liability is a live question.)
